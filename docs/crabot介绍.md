# Crabot · AI 员工 · 电商落地

> 给投资人看的产品设计图 — 2026-05-06

---

## 一句话

**Crabot 是一个面向 B 端的"团队级 AI 员工平台"——不是个人助理（personal assistant）。我们围绕企业落地的真实痛点（多人协作、权限管控、可审计、可交班、可托付）做了一整套工程化能力，让任何主流模型都能在企业场景里干出大厂级的活。第一个落地方向是电商。**

---

## 我们 vs OpenClaw / Hermes 这类 personal AI agent

> OpenClaw、Hermes 这类已经把"个人助理"路线做得很完整：多通道、多模型、子代理、定时、Skill 标准、跨会话记忆 —— 这些能力 Crabot 也有，但**已经是行业基线，不是差异**。
> 我们重点说不一样的部分。

### 根本不同：**他们是 single-user personal assistant，我们是 multi-user enterprise platform**

这一条决定了下面的所有差异。

| | OpenClaw / Hermes | **Crabot** |
|---|------------------|------------|
| **目标用户** | 个人玩家 / 开发者 | 电商团队 / 品牌矩阵 / 代理商 |
| **入口** | CLI + 配置文件 + 个人通道 | **Web 管理后台** + 多通道 + CLI 三位一体 |
| **多人协作** | 单用户为主 | 老板 / 运营 / 客服 多角色 |
| **权限** | 无 | **fail-closed 三级权限模板** |
| **审计** | 日志 | **Trace + Span 树**，每一步可回看可审计 |
| **多租户** | 无 | **一台服务器跑多套独立 Crabot** |
| **配置变更** | 改文件 / 重启 | **Web 改完即时下发到运行中的 Agent，不停机** |
| **行业纵深** | 通用助手 | **电商垂直**：视觉 / 营销 / 供应链 / 运营 |
| **凭证管理** | 用户自己保管 | **Admin OAuth 自动刷新**，运维零负担 |

> 一句话：**他们是给极客用的瑞士军刀，我们是给企业用的生产力车间。**

---

## Crabot 真正不一样的六件事

### 1. 企业级管控层（personal assistant 没有这一层）

- **Web 管理后台** —— 可视化配置模型 / 通道 / 权限 / 工具 / Skill / 调度，不靠 CLI
- **权限模板** —— 老板能干所有事；运营能上架改价不能提现；客服只能查询回复；实习生只读。**AI 越权由系统拦截，不靠模型自觉**
- **任务追溯** —— Trace + Span 树记录每一次决策、每一次工具调用，AI 干错了能精准定位、精准纠正、给客户/法务交代


> 这一层是 6~12 个月的工程量，是我们和 personal AI agent 之间的护城河。

---

### 2. 双层 Agent 架构（Front + Worker）

市面上的 agent 架构基本都是「main agent + 子代理」（Hermes 也是这种）。**Crabot 不是这种结构**。

```
                    用户消息
                       │
                       ▼
              ┌─────────────────┐
              │ Front（分诊员）   │  最多 5 步内决策
              │  cheap & fast   │  ─ 直接回复
              │  小模型即可      │  ─ 派任务给 Worker
              └────────┬────────┘  ─ 追加补充
                       │            ─ 静默观察
              ┌────────┴────────┐
              ▼                 ▼
       ┌────────────┐    ┌──────────────────┐
       │ Worker     │    │ Worker           │
       │ smart 强模型│    │ 长任务跨夜执行     │
       │ 单目标专心  │    │ 自维护待办        │
       └────────────┘    └──────────────────┘
```

**关键工程价值**：
- **Front 用便宜小模型，Worker 用强模型** —— 整体 token 成本下降 60–70%
- **Front 5 步硬上限** —— 不让模型自己跑长链路调查，复杂事一律派 Worker
- **Worker 自维护待办清单**，跨上下文压缩仍能保持目标一致

> 这个架构是从"真公司怎么分工"抽象出来的，不是工程师的奇思妙想。

### 3. Agent 设计哲学：刻意学习并模仿人类员工

这是 Crabot 区别于市面所有 agent 的**精神内核**：我们不把 agent 设计成"会对话的程序"，而是设计成"会用 IM 工作的虚拟员工"。很多看似反直觉的设计选择，背后都是这条原则。注意力调度模块的代码注释里直接写着 "模拟人类查看 IM 消息的注意力模式"——这不是对外的营销话术，是工程内核。

| 设计选择 | 模仿人类的什么行为 | 商业价值 |
|---------|---------------------|----------|
| **不主动留存聊天历史，要看就去 IM channel 翻** | 你不会逐字背微信对话，需要时才翻一下 | agent 上下文不被冗余消息拖垮，token 长期稳定 |
| **决策"派任务"时必须先发一句"好的，我来处理"** | 同事接任务先说"收到"再开干 | 用户不傻等、不焦虑、不疑惑 AI 在不在 |
| **群聊默认旁听不发言，被 @ 才必须回** | 你在群里大部分时候是潜水的 | agent 不刷屏、不抢话、不越界 |
| **注意力调度：刚发言完高频巡检，久未参与拉长间隔** | 你刚发完消息会盯一会儿，然后注意力转走 | 群消息既不漏也不抢，节奏像真人 |
| **短期记忆是"事件流水账"，不存字面对话** | 人脑记的是"昨天 X 告诉我 Y"，不是逐字逐句 | 决策不被措辞带偏、跨 session 仍能调用 |
| **每条消息和工具结果都带时间戳** | 你做事知道"刚才 / 上午 / 昨天" | 跨日跨小时任务不错乱时间感 |
| **Worker 自维护 todo 清单，上下文压缩后仍可见** | 你把待办记便签上，开会回来还能接着做 | 长任务不丢目标、不漂移 |
| **后台进程跨夜跑，第二天交活** | 老板睡前布置任务、第二天来收成果 | 不让用户陪 AI 熬夜，工作流真正异步 |
| **可被随时打断、可追加补充** | 同事汇报到一半你打断说"等等改方向" | 用户随时纠偏，AI 不钻牛角尖 |

**这套哲学的三个副产品**：
- **体验像同事**——用户像和真人沟通，不像在和命令行说话
- **工程上更稳**——人类协作模式经过几千年优化，比工程师从零设计的 agent 工作流可靠得多
- **成本上更省**——「不主动记历史」≈ 不为不必要的上下文付费，token 自然下降

> OpenClaw / Hermes 走的是 **"工具集成"路线** —— agent 是命令调度中心。
> **Crabot 走的是"组织行为"路线 —— agent 是会用 IM 上班的虚拟员工。**

这条路线的另一个意义：**当我们把 agent 设计得越像人，企业把它当员工用就越自然**。培训成本低、上手就懂、出错时凭直觉能纠正。这是 B 端落地的隐性壁垒。

### 4. 多模型协同（不是模型切换）

**Hermes 的做法**：`hermes model` 选一个模型用。
**Crabot 的做法**：**多个模型同时在线协同**，主模型 + 视觉专家 + 推理专家 + 长文本专家各司其职。

```
店主："帮我看一下这张图，写一段适合朋友圈的文案"
              │
              ▼
       主模型（DeepSeek，便宜）做调度判断
              │
              ├──▶ 视觉专家（Qwen-VL / Gemini Vision）看图、出描述
              │              │
              │              ▼
              ◀──── 描述 ─────
              │
              ▼
       主模型基于描述 + 店铺记忆 → 生成文案
```

**店主的体验和直接用 GPT-5 一样，成本是它的 1/10。**

明年 DeepSeek 自己有视觉了？改一行配置切回原生。**模型升级我们立刻吃红利，模型降价我们立刻传给客户。**

### 5. 工程化的记忆与进化系统（比 agent-curated memory 深一层）

Hermes 主打 "self-improving learning loop"，核心机制是**让模型自己 curate memory + 周期性提醒自己**。这个思路对个人玩家够用，对企业不够：模型可能学错、可能学坏、可能学进噪声，没法回滚、没法审计、没法多人共享。

我们做的是**严格工程化的记忆系统**：

| 机制 | 解决什么 |
|------|---------|
| **PE-Gated Write 写入门控** | 不是模型说写就写，必须满足显式条件才落库 |
| **观察期晋升** | 新经验先放观察 → 多次验证才晋升为 SOP，避免一次误学污染 |
| **Case → Rule 演化** | 多次同类纠正自动抽象成规则 |
| **冻结快照 + 版本历史** | 一键回滚到任意历史版本，"学坏了"也能救 |
| **双时态记录（bi-temporal）** | 既知道"事实何时发生"，也知道"何时被记下"，审计完整 |
| **反馈信号 4 档闭环** | 店主每次表扬 / 默认 / 抱怨 / 强烈否决都被精准量化 |
| **Chain-of-Note + RRF + Reranker** | 多通路融合检索，召回更准 |

> 一个能力强的小模型 + 一套工业级记忆系统 = 比一个强模型 + 弱记忆，效果稳定得多。

### 6. 通用底座 + 行业纵深的扩展路径

Crabot 当前是**通用型 AI 员工平台**，底层不和任何单一行业绑定。我们刻意没有把行业逻辑写死在内核里，而是把它放在 Skill / 工具层 —— **底座越通用，行业适配越快**。

电商是我们选定的首个商业化方向，因为底座的核心能力栈高度契合电商的真实痛点：

| 电商真实痛点 | Crabot 底座对应能力 |
|--------------|---------------------|
| 商家自己没视觉模型，但又要批量出图 | **多模型协同** —— 主模型 + 视觉专家组合 |
| 每家店有自己的语气、品控、SOP | **工程化记忆系统** —— 沉淀店铺 know-how |
| 老板/运营/客服权限要分层 | **三级权限模板** —— fail-closed 不越权 |
| AI 改了价上错了货，老板要追责 | **Trace 追溯** —— 每一步可回看可审计 |
| 大促期长任务跨夜跑 | **后台进程 + 任务调度** |
| 总账号要管多家店 | **多实例隔离** —— 一台服务器多租户 |

落地方式：**用底座原生能力 + 围绕电商场景沉淀 Skill 包和工具集成**（抖店 / 淘宝 / Shopify / ERP 等），把通用平台快速塑造成"开箱即用的电商版 Crabot"。

电商验证后，同样的底座可以按这个路径切入其他垂直行业（教育、专业服务、品牌私域……）。**这是"通用底座 + 行业模板"的双层资产结构 —— 底座是技术壁垒，模板是行业飞轮。**

---

## 我们的壁垒

1. **B 端企业管控全栈** —— Web 后台 / 权限 / 审计 / 多租户 / 不停机配置 / OAuth 自动刷新，是 personal agent 6~12 个月也补不齐的工程量
2. **双层 Agent 架构** —— Front+Worker 的 token 经济性 + 长任务稳定性，是从"真公司分工"抽象出来的设计
3. **工程化记忆系统** —— 比 agent-curated memory 深一层，可控、可审计、可回滚，**企业敢上的关键**
4. **多模型协同（不是切换）** —— 让小模型补短板、吃红利
5. **通用底座的行业可塑性** —— 通用底座 + 行业 Skill 包的双层资产结构，**第一个行业验证后，下一个行业切入成本指数下降**
